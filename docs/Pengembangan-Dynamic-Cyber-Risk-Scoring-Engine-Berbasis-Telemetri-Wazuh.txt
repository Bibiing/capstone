# Pengembangan Dynamic Cyber Risk Scoring Engine Berbasis Telemetri Wazuh Studi Kasus Simulasi Perbankan Daerah
# CAPSTONE PROJECT KOLABORASI INDUSTRI – ITS 2026

## Latar Belakang
Security Operations Center (SOC) modern menghasilkan ribuan alert dan log setiap harinya. Namun, volume data yang besar ini justru
menciptakan tantangan baru: bagaimana menerjemahkan sinyal teknis menjadi informasi risiko yang bermakna bagi pengambil keputusan?

Tantangan SOC
- Volume alert sangat tinggi, sulit diprioritaskan secara manual
- Bahasa teknis sulit dipahami manajemen non-teknis
- Tidak ada scoring risiko terpadu per aset

Kebutuhan Manajemen
- Prioritas mitigasi berbasis risiko nyata
- Pemantauan tren risiko secara periodik
- Ringkasan eksekutif untuk keputusan cepat dan tepat

## Problem Statement 
Inti permasalahan yang ingin diselesaikan proyek ini adalah: bagaimana mengubah telemetri Wazuh — berupa alert, log, dan data kerentanan — menjadi informasi risiko yang dapat digunakan secara langsung?

1. Skor Risiko Dinamis
Setiap aset (server, workstation, aplikasi) mendapatkan skor risiko yang diperbarui secara periodik berdasarkan telemetri terkini.

2. Ringkasan Eksekutif
Menyajikan indeks risiko global dan area risiko tertinggi dalam format yang mudah dipahami oleh CISO dan manajemen.

3. Insight yang Actionable
Menghasilkan daftar prioritas mitigasi yang jelas — bukan sekadar angka, melainkan rekomendasi langkah konkret berikutnya.

## Tujuan Proyek
Proyek ini bertujuan membangun Proof of Concept (PoC) Risk Scoring Engine berbasis telemetri Wazuh dalam lingkungan simulasi
perbankan daerah. Output utama yang diharapkan mencakup empat komponen berikut:

🔬 PoC Engine
Membangun Risk Scoring Engine berbasis data simulasi Wazuh yang dapat berjalan end-to-end.

📊 Skor Dinamis
Menghasilkan skor risiko per aset (server/workstation/app) yang diperbarui secara berkala.

🖥️ Dashboard Risiko
Menyediakan tampilan ranking aset, tren risiko, dan ringkasan untuk pengambil keputusan.

⚙️ Model Transparan
Scoring berbasis rule-based yang explainable — setiap skor dapat dijelaskan faktor penyebabnya.


## Scope: In-Scope
Berikut adalah komponen yang secara eksplisit masuk dalam lingkup pengerjaan proyek ini, mencakup tiga lapisan utama: ingesti data,
mesin scoring, dan visualisasi.

1. Ingesti Telemetri
- Wazuh alert (severity, rule, agent)
- Vulnerability summary (CVSS agregat)

2. Risk Scoring Engine

3. Dashboard & Simulasi
- Top risk assets ranking
- Risk trend visualization
- Drill-down penyebab skor
- Simulasi spike event & remediation decay

## Use Case Utama
Sistem dirancang untuk mendukung enam skenario penggunaan kritis yang menjawab kebutuhan nyata tim keamanan dan manajemen
perbankan daerah.

1. Risk Score per Asset
Skor 0–100 atau level L/M/H untuk setiap aset terdaftar.

2. Risk Breakdown
Dekomposisi skor ke faktor: ancaman, kerentanan, dan kritikalitas aset.

3. Risk Trend
Pemantauan tren risiko harian dan mingguan untuk deteksi dini perubahan postur keamanan.

4. Risk Spike Detection
Deteksi kenaikan skor risiko signifikan sebagai indikator awal insiden.

5. Executive View
Indeks risiko global dan ringkasan area risiko tertinggi untuk lapisan manajemen.

6. Prioritization List
Rekomendasi urutan mitigasi berdasarkan skor risiko tertinggi dan dampak potensial.

## User Story — Perspektif CISO
Berikut contoh user story yang menjadi dasar perancangan fitur sistem, ditulis dari perspektif Chief Information Security Officer (CISO)
sebagai pengguna utama executive view.

User Story
Sebagai CISO, saya ingin melihat skor risiko per aset secara real-time, agar saya dapat menentukan prioritas mitigasi dengan cepat dan berbasis data — tanpa harus membaca ratusan alert teknis secara manual.

Acceptance Criteria
- Tampil daftar Top 5–10 risk assets dengan skor terkini
- Skor diperbarui secara periodik (per jam atau per hari)
- Klik pada aset → tampil breakdown penyebab skor (threat, vuln, criticality)
- Tersedia tampilan tren risiko dalam periode tertentu

## MVP Definition — Kriteria Keberhasilan PoC
PoC dinyatakan berhasil apabila keempat kondisi berikut terpenuhi secara end-to-end pada sprint akhir:

Data Masuk ✅
Alert Wazuh berhasil diingesti — baik via replay dataset maupun Wazuh lab instance yang aktif.

Skor Dihitung ✅
Risk scoring engine menghasilkan skor per aset secara otomatis berdasarkan data yang masuk.

Dashboard Aktif ✅
Dashboard menampilkan ranking aset, tren risiko, dan drill-down penyebab skor dengan jelas.

Simulasi Terlihat ✅
Simulasi spike meningkatkan skor secara visible di dashboard, dan skor turun setelah remediation.

## Simulasi Input Wazuh — Pendekatan Dataset
Karena tidak menggunakan data produksi, proyek mengandalkan pendekatan simulasi berlapis. Dataset dirancang untuk mencakup empat
skenario kritis yang merepresentasikan kondisi nyata di SOC perbankan.

Tahapan Pendekatan

1. Tahap Awal (Cepat)
Dataset replay dalam format JSON/CSV yang menyerupai struktur Wazuh alert — memungkinkan pengembangan paralel tanpa infrastruktur.

2. Tahap Lanjutan (Opsional)
Wazuh lab instance aktif dengan dummy agents untuk menghasilkan telemetri yang lebih realistis dan dinamis.

Skenario Dataset Wajib
- Normal: Aktivitas harian tanpa anomali signifikan
- Spike / Incident Burst: Lonjakan alert simulasi insiden
- Vulnerability Cluster: Konsentrasi CVE pada aset tertentu
- Remediation + Decay: Skor menurun setelah perbaikan

## Data Contract — Struktur Field Minimal
Data contract mendefinisikan format minimum yang wajib dipenuhi oleh setiap event input maupun output scoring. Ini adalah fondasi interoperabilitas seluruh komponen sistem.

### Input Event (Telemetri Wazuh)
Field, Deskripsi
timestamp, Waktu kejadian event
asset_id / hostname, Identifikasi unik aset
severity, Level dari Wazuh rule (1–15)
category, auth / malware / integrity / network
event_type, alert / vuln / control

### Output Scoring (Risk Engine)
Field, Deskripsi
asset_id, Identifikasi aset yang dinilai
risk_score, Skor risiko agregat (0–100)
risk_components, Breakdown: threat / vuln / criticality
timestamp, Waktu perhitungan skor

Data contract harus dikunci di Sprint 1 sebelum pengembangan engine dan dashboard dimulai, untuk memastikan konsistensi antar komponen tim.

## Risiko Utama & Strategi Mitigasi
Tiga risiko berikut diidentifikasi sebagai ancaman terbesar terhadap keberhasilan proyek. Setiap risiko disertai strategi mitigasi konkret yang
diterapkan sejak awal pengerjaan.

1. ⚠️ Dataset Tidak Realistis
Risiko: Simulasi terlalu sederhana scoring tidak meaningful sulit divalidasi relevansinya.
Mitigasi: Kunci data contract dari Sprint 1 dengan skenario lengkap (normal, spike, vuln, remediation).

2. ⚠️ Over-Engineering
Risiko: Kompleksitas teknis berlebihan (ML/AI prematur) → timeline meleset → MVP tidak tercapai.
Mitigasi: Gunakan rule-based scoring yang explainable. Simpan kompleksitas sebagai backlog fase berikutnya.

3. ⚠️ Dashboard Terlalu Teknis
Risiko: Output tidak dapat digunakan oleh manajemen non-teknis → kehilangan nilai bisnis.
Mitigasi: Review industri per sprint untuk memastikan tampilan dan bahasa sesuai kebutuhan eksekutif.
